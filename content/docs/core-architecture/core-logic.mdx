# Архитектура и Ядро (`mu-plugins`)

Этот раздел описывает фундамент бизнес-логики нашего Boilerplate. Все базовые функции (безопасность, производительность и маршрутизация хуков) находятся в директории `mu-plugins` (Must-Use Plugins). 

<Callout type="info">
  **Почему Must-Use Plugins?** 
  Эти плагины загружаются ядром WordPress автоматически до инициализации темы и любых обычных плагинов. Их **невозможно отключить из админки**, что надежно защищает сайт от "случайных поломок" контент-менеджерами или клиентами.
</Callout>

---

## 1. Главный загрузчик (`agency-core-loader.php`)

Мы **не пишем бизнес-логику** (Custom Post Types, Таксономии, обработку форм) в `functions.php` темы! Тема отвечает исключительно за визуал. Вся бизнес-логика живет в папке `mu-plugins/agency-b2b-core/`. Чтобы WordPress увидел эту папку, мы используем загрузчик `agency-core-loader.php`.

### Порядок загрузки
Загрузчик использует жесткий массив `$load_order`, чтобы предотвратить конфликты зависимостей:

```php
private array $load_order = [
    'security',
    'taxonomies',  // Таксономии всегда регистрируются ДО CPT!
    'post-types',
    'custom-fields',
    'queries',
    'ajax',
    'integrations',
    'scripts',
    'admin',
];
```

<Callout type="warning">
  **Почему важен порядок?** Если вы зарегистрируете Custom Post Type до таксономии, то при сохранении поста таксономия может не сохраниться или отработать некорректно в фильтрах. Жесткий порядок исключает эту проблему на уровне архитектуры.
</Callout>

### Фикс пермалинков
Также в загрузчике есть глобальный фикс для сброса пермалинков при смене темы (избавляет от необходимости вручную пересохранять постоянные ссылки):
```php
// Centralized flush rewrite rules to avoid multiple database calls
add_action( 'after_switch_theme', 'flush_rewrite_rules' );
```

## 2. Безопасность: Ограничитель входов (`ag-login-limiter.php`)

Вместо установки громоздких сторонних плагинов (вроде Limit Login Attempts), мы написали свой легковесный и пуленепробиваемый скрипт для защиты от брутфорс-атак на `wp-login.php`.

### Как это работает:
Скрипт перехватывает попытки авторизации. Если пользователь ввел неправильный пароль **4 раза подряд**, его IP блокируется на 20 минут. Данные хранятся во временном кэше WordPress (Transients API).

```php
// Увеличить счётчик при неудачном входе
public static function on_failed($username) {
    $data = self::get_data(); // Получаем данные по IP
    $data['attempts']++;

    if ($data['attempts'] >= self::MAX_ATTEMPTS) {
        $data['locked_until'] = time() + self::LOCKOUT_TIME; // Бан на 20 минут
        $data['attempts'] = 0; // Сбрасываем попытки после бана
    }

    // Сохраняем в кэш
    set_transient(self::get_key(), $data, self::RESET_TIME);
}
```

<Callout type="error" title="Архитектурный плюс">
  Проверка блокировки происходит на хуке `wp_authenticate_user` — **до того**, как WP полезет в базу сверять пароль. Если IP в бане, скрипт мгновенно возвращает `WP_Error`, обрывая ресурсоемкие запросы к БД.
</Callout>

## 3. Производительность: Конвертер WebP (`ag-webp-converter.php`)

Мы отказались от тяжелых плагинов конвертации (вроде Converter for Media) в пользу собственного скрипта. Он автоматически генерирует `.webp` копии загружаемых изображений без ущерба для скорости.

### Генерация при загрузке
При загрузке картинки срабатывает стандартный хук WP `wp_generate_attachment_metadata`. Наш скрипт "вклинивается" в этот процесс и создает `.webp` копии для оригинала и **для всех сгенерированных WP миниатюр**.

Конвертер использует нативные PHP библиотеки (`imagecreatefromjpeg`, `imagewebp`), заботясь об альфа-канале для PNG:
```php
case 'image/png':
    $image = @imagecreatefrompng($file_path);
    if ($image) {
        imagepalettetotruecolor($image);
        imagealphablending($image, true);
        imagesavealpha($image, true); // Сохраняем прозрачность
    }
    break;
```

### Очистка "мусора"
При удалении картинки из медиабиблиотеки (хук `delete_attachment`), наш скрипт также удаляет все созданные `.webp` копии. Сервер не забивается "осиротевшими" файлами.

### Подмена на фронтенде (Nginx Bypass)
Как браузер узнает о существовании WebP? 
Плагин через хук `template_redirect` и буферизацию вывода (`ob_start`) ищет в финальном HTML-коде все картинки из папки `uploads` и добавляет к ним суффикс `-optimized`:

```php
// Заменяет https://site.ru/wp-content/uploads/image.jpg 
// на https://site.ru/wp-content/uploads/image-optimized.jpg
return $base_path . '-optimized.' . $ext;
```

Запрос с суффиксом `-optimized` уходит на сервер, а `.htaccess` (как описано в разделе [Инфраструктура и CI/CD](/docs/workflow/infrastructure-cicd)) решает: отдать WebP (если браузер поддерживает) или оригинальный JPG. Это работает молниеносно и полностью обходит проблемы с кэшированием Nginx.
