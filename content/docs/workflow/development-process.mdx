# Архитектура и Процесс Разработки Бойлерплейта

Разработка на базе нашего бойлерплейта строится на жестком разделении клиентского кода и кода "Ядра" (Core). Это необходимо для того, чтобы новые фичи и багфиксы могли доставляться во все старые клиентские проекты без боли и конфликтов при слиянии.

## Главная боль: Почему Git Upstream вызывает конфликты?

Представьте ситуацию: вы запустили проект клиента из бойлерплейта, клиенту не нужны "Кейсы", и вы просто удалили файл `cpt-cases.php`.
Через месяц в ядре бойлерплейта выходит важное обновление безопасности для `cpt-cases.php`. 
Git отслеживает строки кода. Когда вы притянете обновления (`git fetch upstream`), Git выдаст конфликт: *"В источнике этот файл обновили, а тут его кто-то стер"*.

**Правило №1:** Код ядра (Boilerplate Core) **нельзя модифицировать или удалять** в клиентских проектах.

---

## Решение: Модульность и Feature Toggles

Чтобы избежать конфликтов, мы переходим от парадигмы *"Удалить лишнее"* к парадигме *"Включить нужное"*.

Бойлерплейт содержит в себе **всё** (все CPT, все блоки, все интеграции). Но по умолчанию они выключены. 
Если клиенту не нужны "Кейсы", вы **не удаляете** код кейсов из `mu-plugins/agency-b2b-core`. Вы просто "не нажимаете кнопку включения".

<Tabs items={['Вариант 1: add_theme_support', 'Вариант 2: wp-config.php']}>
  <Tab value="Вариант 1: add_theme_support">
    Тема клиента решает, какие фичи ядра ей нужны. Это рекомендуемый WordPress-way.
    
    В клиентской теме (`functions.php`):
    ```php
    // Включаем нужные фичи ядра для ЭТОГО клиента
    add_action('after_setup_theme', function() {
        add_theme_support('agency-cpt-services');
        add_theme_support('agency-integration-amocrm');
        
        // add_theme_support('agency-cpt-cases'); // Мы просто не просим ядро регистрировать кейсы
    });
    ```
    
    В ядре (`mu-plugins/agency-b2b-core/core.php`):
    ```php
    add_action('init', function() {
        if (current_theme_supports('agency-cpt-cases')) {
            require_once __DIR__ . '/includes/cpt-cases.php';
        }
    });
    ```
  </Tab>
  <Tab value="Вариант 2: wp-config.php">
    Подходит для жестких системных интеграций на уровне сервера.
    
    В `wp-config.php` клиента:
    ```php
    define('AGENCY_ENABLE_CASES', false);
    ```
    
    В ядре `agency-b2b-core`:
    ```php
    if (defined('AGENCY_ENABLE_CASES') && AGENCY_ENABLE_CASES === true) {
        require_once __DIR__ . '/includes/cpt-cases.php';
    }
    ```
  </Tab>
</Tabs>

## Идеальный процесс разработки для Агентства

<Steps>
  <Step title="1. Развитие Бойлерплейта (upstream)">
    Вы придумываете новые фичи (новые CPT, блоки, компоненты). Все новые фичи сразу оборачиваются в Feature Toggles (условия проверки). Вы коммитите это в главный репозиторий.
  </Step>
  <Step title="2. Старт нового проекта (origin)">
    Вы делаете `git clone` бойлерплейта. Переименовываете `origin` в `upstream` и добавляете новый клиентский `origin`. Заходите в конфигурацию и **включаете** только то, что нужно конкретному клиенту.
  </Step>
  <Step title="3. Обновление старого клиента (Магия без конфликтов)">
    Прошел год. Вы делаете `git fetch upstream && git merge upstream/main` в проекте клиента. Так как вы **не удаляли** файлы ядра, слияние проходит без единого конфликта. Новая фича "Отзывы" приехала, но по умолчанию скрыта за Feature Toggle. Сайт клиента продолжает работать как часы.
  </Step>
</Steps>

## Итог: Правило "Святого Ядра"

<Callout type="error" title="Строго запрещено">
  * **Никогда** не правьте файлы `agency-b2b-core` в проекте клиента (делайте это в бойлерплейте и тяните апдейт).
  * **Никогда** не удаляйте файлы из `agency-b2b-core` в проекте клиента.
</Callout>

### Как выстроить процесс без оверинжинеринга на практике

1. **Разделяй и властвуй на уровне папок:**
   * `mu-plugins/agency-b2b-core/` — "Святая земля" бойлерплейта. Обновляется из Git. Изменять нельзя.
   * `mu-plugins/client-core.php` — Земля клиента. Здесь пишется любая уникальная бизнес-логика клиента.

2. **Фильтры вместо Delete (Удаление полей):**
   Если вам нужно убрать мета-поле (например, поле "Бюджет" у Кейсов), не удаляйте его из ядра. Используйте фильтры WordPress в клиентском плагине:
   
   ```php
   // В mu-plugins/client-core.php:
   add_filter('agency_cases_meta_fields', function($fields) {
       // Отфильтровываем поле по имени
       return array_filter($fields, fn($f) => $f->get_name() !== 'budget');
   });
   ```

3. **Сознательный отказ от обновлений UI:**
   Как только вы начали верстать уникальный дизайн для клиента, тема больше не нуждается в обновлениях "универсальных" стилей. Вытягивайте из `upstream` только обновления бэкенда (`agency-b2b-core`), но не трогайте уникальную CSS верстку клиента (можно использовать `git cherry-pick`).
