# Инфраструктура и CI/CD

В этом документе описывается локальное окружение (Docker), управление зависимостями, пайплайн автоматического развертывания (GitHub Actions) и серверные настройки для оптимизации отдачи графики.

---

## Управление зависимостями (Composer)

Вместо того чтобы скачивать плагины вручную или заливать их в Git-репозиторий (что нарушает Enterprise-стандарты), мы используем **Composer**.

Файл `composer.json` в корне проекта управляет PHP-зависимостями:
- `htmlburger/carbon-fields` — основа нашей гибридной архитектуры данных.
- `enshrined/svg-sanitize` — библиотека для безопасной загрузки SVG.

При развертывании проекта достаточно выполнить `composer install`, и все необходимые библиотеки будут загружены в папку `vendor/`.

## Локальная разработка (Docker)

Для локальной разработки мы используем Docker. Это гарантирует, что у всех разработчиков в команде будет абсолютно одинаковая версия PHP, MySQL и одинаковые настройки сервера.

<Tabs items={['Быстрый Docker (Рекомендуемый)', 'Полный Docker']}>
  <Tab value="Быстрый Docker (Рекомендуемый)">
    ### `docker-compose.light.yml` (Для Windows/Mac)
    
    Главная проблема Docker на Windows и Mac — это очень медленная работа I/O при монтировании больших объемов файлов. Монтирование всего ядра WordPress (~6000 файлов) заставляет сайт грузиться по 5-10 секунд локально.
    
    В `light` конфиге мы **не монтируем ядро WordPress**. Контейнер берет чистое ядро прямо из Linux-образа, а мы пробрасываем только те папки, в которых пишем код:
    
    ```yaml
    volumes:
      - ./themes/:/var/www/html/wp-content/themes
      - ./vendor/:/var/www/html/wp-content/vendor
      # Пробрасываем только картинки, чтобы сайт выглядел нормально
      - ./.srv/wordpress/wp-content/uploads:/var/www/html/wp-content/uploads
    ```
    
    *Замечание:* Если вам нужно разрабатывать логику ядра, добавьте строку `- ./mu-plugins/:/var/www/html/wp-content/mu-plugins` в свой локальный конфиг.
  </Tab>
  <Tab value="Полный Docker">
    ### `docker-compose.yml`
    
    Полный конфиг монтирует всю папку WordPress локально (`./.srv/wordpress`). Используйте его только если вы работаете на Linux, где нет проблем с производительностью файловой системы, или если вам нужно жестко дебажить ядро самого WordPress.
  </Tab>
</Tabs>

## Обход кэширования WebP (Конфигурация сервера)

Одной из главных фишек бойлерплейта является встроенный быстрый конвертер WebP (плагин `ag-webp-converter.php`). Чтобы он работал молниеносно, мы обходим PHP и Nginx, отдавая изображения прямо через `.htaccess`.

<Accordions>
<Accordion title="Посмотреть конфигурацию .htaccess">
```apache
<IfModule mod_rewrite.c>
  RewriteEngine On

  # Шаг 1: Убираем суффикс -optimized (который добавляется нашим плагином в HTML)
  RewriteCond %{REQUEST_FILENAME} !-f
  RewriteRule ^(.+)-optimized\.(jpe?g|png)$ $1.$2 [L]
  
  # Шаг 2: Если браузер поддерживает WebP и файл существует, отдаем .webp вместо оригинала
  RewriteCond %{HTTP_ACCEPT} image/webp
  RewriteCond %{REQUEST_FILENAME} -f
  RewriteCond %{REQUEST_FILENAME}.webp -f
  RewriteRule ^(.*)\.(jpe?g|png)$ $1.$2.webp [T=image/webp,E=ACCEPT:1,L]
</IfModule>
```
</Accordion>
</Accordions>

<Callout type="info">
  **Зачем это нужно:** Вместо того чтобы запускать тяжелый PHP-скрипт при каждом запросе картинки (как это делают обычные плагины оптимизации), сервер Apache сам на лету подменяет картинку. Это экономит огромное количество ресурсов сервера (CPU и RAM).
</Callout>

## CI/CD: Автоматический Деплой (GitHub Actions)

Вам **никогда** не нужно загружать файлы на сервер по FTP вручную. За нас это делает GitHub Actions. Пайплайн настроен в файле `.github/workflows/CICD.yml`.

### Триггеры (Когда срабатывает деплой)

- **DEV-сервер:** Деплой происходит автоматически при каждом `push` в ветку `dev`.
- **PROD-сервер:** Деплой происходит только при создании **Release** (релиза) в интерфейсе GitHub.

### Этапы работы пайплайна

<Steps>
  <Step title="Сборка фронтенда">
    Поднимается контейнер с Node.js. Запускаются скрипты сборки и минификации темы (`npm run build`).
  </Step>
  <Step title="Подготовка архивов">
    Скачивается чистое ядро WordPress. Создается архив `wpcontent.zip`, куда пакуются `plugins`, `mu-plugins`, `themes` и `vendor`. Директория `node_modules` **исключается**, чтобы не тащить мусор на сервер!
  </Step>
  <Step title="Безопасный Webhook">
    Генерируется уникальный 16-значный токен. Он вшивается в специальный скрипт распаковки `deploy.php`.
  </Step>
  <Step title="FTP Загрузка (lftp)">
    Через защищенное FTP-соединение заливаются архивы, `deploy.php` и файл `.maintenance` (который временно закрывает сайт "заглушкой", чтобы никто не увидел поломанный интерфейс в момент распаковки).
  </Step>
  <Step title="Распаковка">
    GitHub дергает Webhook по секретному URL. Сервер распаковывает архивы, удаляет `.maintenance` и самоудаляет скрипт `deploy.php`.
  </Step>
</Steps>

### Секреты (GitHub Secrets)

Для развертывания пайплайна на новом проекте, добавьте следующие ключи в настройки репозитория (Settings -> Secrets and variables -> Actions):

*   `SSH_USER`: Логин от FTP (например, `u1234567`).
*   `SSH_PASSWORD`: Пароль от FTP.
*   `SSH_TARGET_DEV`: Путь на сервере к папке dev-сайта (например, `/dev.domain.ru/public_html/`).
*   `SSH_TARGET_PROD`: Путь на сервере к боевому сайту.
*   `SITE_URL_DEV`: URL тестового сайта.
*   `SITE_URL_PROD`: URL боевого сайта.
