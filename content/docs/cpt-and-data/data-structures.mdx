# Структуры данных (CPT, Таксономии, Carbon Fields)

В этом разделе мы разбираем, как данные регистрируются, хранятся и управляются в админке WordPress. Наш Boilerplate использует парадигму разделения контента на "Структурированный" (Structured CPT) и "Повествовательный" (Narrative CPT).

Вся регистрация структур данных происходит на уровне ядра (`mu-plugins/agency-b2b-core/`).

---

## 1. Регистрация CPT (Custom Post Types)

Все типы записей регистрируются в папке `post-types/`. Рассмотрим на примере Каталога (`catalog.php`).

### Размещение в меню
Параметр `'menu_position' => 5` и `'menu_icon' => 'dashicons-cart'` помещают раздел "Каталог" сразу под стандартным разделом "Записи" в боковом меню админки.

### Ключевое правило архитектуры (Structured CPT)

Для структурированных данных (таких как Каталог товаров или Кейсы) мы жестко отключаем визуальный редактор. 

```php
$product_args = array(
    // ... стандартные лейблы и параметры
    'rewrite'      => array( 'slug' => 'catalog' ),
    'has_archive'  => true,
    'supports'     => array( 'title', 'thumbnail', 'excerpt', 'revisions' ),
    'show_in_rest' => false, // 🔥 КРИТИЧЕСКИ ВАЖНО 🔥
);
register_post_type( 'catalog_product', $product_args );
```

<Callout type="warning">
  **Почему `'show_in_rest' => false`?** Это отключает блочный редактор Gutenberg для карточек товаров. Зачем? Чтобы контент-менеджер не мог разрушить БЭМ-верстку карточки товара, добавляя туда случайные абзацы или нестандартные кнопки. Товар рендерится *строго из мета-полей*.
</Callout>

Для **Narrative CPT** (Услуги, Лендинги) мы устанавливаем `'show_in_rest' => true`, так как каждая услуга — это уникальный лонгрид, который собирается из изолированных блоков Gutenberg.

## 2. Таксономии (Категории, Метки, Бренды)

Таксономии регистрируются в `taxonomies/`. На примере `catalog-taxonomies.php` мы регистрируем 3 таксономии для `catalog_product`: `catalog_category` (Категории), `catalog_tag` (Метки/Скидки), `catalog_brand` (Бренды).

```php
$category_args = array(
    'hierarchical'      => true, // Как рубрики (с вложенностью)
    'show_in_rest'      => true, // Разрешаем REST API для работы AJAX фильтров на фронте
    'rewrite'           => array( 'slug' => 'product-category' ),
);
register_taxonomy( 'catalog_category', array( 'catalog_product' ), $category_args );
```

## 3. Управление мета-полями (Carbon Fields)

Carbon Fields (`custom-fields/`) — наш основной инструмент для создания полей. Он подключается через Composer и инициализируется на уровне ядра. Разберем сложный файл `catalog-meta.php`.

### 3.1. Создание вкладок в админке
Чтобы редактору было удобно, поля группируются по табам (`add_tab`):

```php
Container::make( 'post_meta', 'Параметры товара' )
    ->where( 'post_type', '=', 'catalog_product' )
    ->add_tab( 'Основное', array(
        Field::make( 'text', 'catalog_sku', 'Артикул' ),
        Field::make( 'text', 'catalog_price', 'Цена' )
            ->set_attribute( 'type', 'number' ),
    ) )
    ->add_tab( 'Галерея', array(
        Field::make( 'media_gallery', 'catalog_gallery', 'Дополнительные фото' )
            ->set_type( array( 'image' ) ),
    ) );
```

### 3.2. Комплексные поля (Повторяющиеся данные)
Для создания таблицы характеристик (где может быть бесконечное число строк), мы используем `complex` поле:

```php
Field::make( 'complex', 'catalog_attributes', 'Характеристики' )
    ->set_layout( 'tabbed-horizontal' )
    ->set_header_template( '<%- attr_name %>' ) // Заголовок свернутого блока
    ->add_fields( array(
        Field::make( 'text', 'attr_name', 'Название параметра' )->set_width( 50 ),
        Field::make( 'text', 'attr_value', 'Значение параметра' )->set_width( 50 ),
    ) );
```
На фронтенде мы получаем это как массив массивов и выводим через обычный цикл `foreach`.

## 4. Бизнес-логика при сохранении (Инварианты)

Carbon Fields позволяет повесить хук на сохранение данных: `carbon_fields_post_meta_container_saved`. Мы используем это для валидации бизнес-правил на стороне сервера.

### Валидация цен
Правило: *"Скидочная цена не может быть больше или равна базовой"*.

```php
$regular_price = (float) carbon_get_post_meta( $post_id, 'catalog_price' );
$sale_price    = carbon_get_post_meta( $post_id, 'catalog_sale_price' );

if ( $sale_price !== '' && (float) $sale_price >= $regular_price ) {
    // Нарушен инвариант! Сбрасываем скидку.
    carbon_set_post_meta( $post_id, 'catalog_sale_price', '' );
}
```

### Автоматическое присвоение бейджей
В этом же хуке скрипт автоматически присваивает товару тег "Скидка", если задана скидочная цена. Контент-менеджеру не нужно вручную ставить галочку!

## 5. Клиентская валидация (JS в админке)

Чтобы улучшить UX контент-менеджера и предотвратить ошибки до отправки формы, мы добавили JS-скрипт (загружаемый через хук `admin_footer-post.php`), который работает "на лету":

<Steps>
  <Step title="Проверка артикула (AJAX)">
    При вводе артикула скрипт шлет AJAX-запрос `wp_ajax_check_catalog_sku`. Если артикул занят другим товаром — поле подсвечивается красным, а кнопка "Опубликовать" блокируется.
  </Step>
  <Step title="Валидация цен (UI)">
    Если редактор пишет скидку больше базовой цены, поле тут же краснеет, кнопка "Опубликовать" блокируется, а при попытке сохранить форму всплывает предупреждение.
  </Step>
  <Step title="Подсказки форматирования">
    Скрипт на лету разбивает миллионы на разряды (`1 000 000 руб.`), чтобы редактор визуально контролировал нули при вводе крупных сумм.
  </Step>
</Steps>
