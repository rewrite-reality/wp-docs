# Концепция и структура документации Boilerplate (Fumadocs)

## 🎯 Цель документации
Документация должна стать единым источником истины (Single Source of Truth) для разработчиков (твоего друга/сотрудника), работающих с WordPress Boilerplate. Она должна быть написана "без воды", с упором на практику, архитектурные ограничения и Git-workflow, чтобы избежать конфликтов при обновлениях.

## 🧱 Основные принципы (на основе промптов)
Перед тем как писать саму доку, нам нужно зафиксировать правила, которые мы извлекли из твоих файлов:
1. **Святое ядро (`mu-plugins/agency-b2b-core/`)** — файлы ядра не удаляем и не правим напрямую. Только включаем/выключаем через *Feature Toggles*.
2. **Гибридная модель CPT** — Narrative CPT (Gutenberg) vs Structured CPT (Carbon Fields).
3. **Изоляция визуала** — тема (`agency-starter-theme`) отвечает только за рендер и стили. Бизнес-логики там нет.
4. **CSS Токены** — жесткая трехуровневая система переменных без хардкода.
5. **Git-Workflow** — `upstream` (бойлерплейт) -> `origin` (клиентский проект).

## 📂 Предлагаемая структура (Разделы для Fumadocs)

Структуру внутри `content/docs` лучше всего разделить на логические модули, от простого к сложному.

### 1. Введение и Быстрый старт (`/docs/getting-started`)
* **Что такое наш Boilerplate?** — Философия, гибридная архитектура, отказ от ACF в пользу Carbon Fields.
* **Установка и Git Workflow** — Как правильно клонировать репозиторий, настроить `upstream` и `origin`, чтобы не было боли при слияниях.
* **Структура папок** — Что где лежит (`mu-plugins` vs `theme`).

### 2. Архитектура и Ядро (`/docs/core-architecture`)
* **Правило "Святого ядра"** — Почему нельзя удалять файлы из `agency-b2b-core` и как это работает.
* **Feature Toggles** — Как включать и выключать функционал через `add_theme_support()` в теме и константы в `wp-config.php`.
* **Клиентский код (`client-core.php`)** — Как и где писать уникальную бизнес-логику клиента и использовать фильтры вместо удаления (например, скрытие полей Carbon Fields).

### 3. Работа с Контентом (CPT & Fields) (`/docs/cpt-and-data`)
* **Narrative CPT (Услуги, Страницы)** — Как работать с атомарными блоками Gutenberg и Block Templates.
* **Structured CPT (Кейсы, Товары)** — Как добавлять и редактировать жесткие поля через **Carbon Fields**. Отключение Gutenberg для таких типов постов.
* **Как добавить новый CPT?** — Пошаговая инструкция добавления кастомного типа записи в проект клиента.

### 4. Визуальный слой и Gutenberg (`/docs/frontend-and-blocks`)
* **CSS и Токены (Design System)** — Трехуровневая система (Primitives, Semantics, Global Layout). Правила использования фоллбэков (`var(--btn-radius, var(--radius-interactive))`).
* **Верстка по БЭМ** — Запрет визуальных модификаторов и хардкода.
* **Gutenberg Блоки** — Как создать свой блок, как стилизовать. Использование `wp_enqueue_block_style` для загрузки "on demand".
* **JavaScript** — Vanilla JS (ES6+), классы, полный отказ от jQuery.

### 5. Регламенты и Чек-листы (`/docs/workflow`)
* **Деплой и Обновление** — Инструкция по получению обновлений ядра из `upstream` без конфликтов. Что делать, если прилетела новая фича? (Она скрыта за Toggle).
* **Чек-лист перед коммитом** — Экранирование (`esc_html`), типизация PHP 8.1+, проверка неймспейсов.

## 🛠 Как будем переписывать (План действий)
1. Снесем текущие "военные" папки (`communications`, `military`) из `content/docs`.
2. Создадим файлы и папки согласно структуре выше.
3. Наполним страницы MDX-контентом, используя выжимки из `development-process.md` и `global-prompt.md`.
4. Настроим `meta.json` файлы Fumadocs для красивой навигации (sidebar).
