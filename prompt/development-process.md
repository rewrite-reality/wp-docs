# Архитектура и Процесс Разработки Бойлерплейта

## Главная боль: Почему Git Upstream вызывает конфликты?

Проблема, с которой вы столкнулись ("удалил CPT кейсов -> притянул обновления -> получил конфликт"), возникает из-за нарушения главного правила использования бойлерплейтов: **Код ядра (Boilerplate Core) нельзя модифицировать или удалять в клиентских проектах.**

Git отслеживает строки кода. Если в бойлерплейте есть файл `cpt-cases.php`, а в проекте клиента вы его удалили, то при следующем обновлении бойлерплейта (если там были правки в `cpt-cases.php`), Git выдаст ошибку: *"В источнике этот файл обновили, а тут его кто-то стер. Конфликт!"*.

## Решение: Модульность и Feature Toggles (Флаги функций)

Чтобы этого избежать, мы переходим от парадигмы *"Удалить лишнее"* к парадигме *"Включить нужное"*.

Ваш бойлерплейт должен содержать в себе **всё** (все CPT, все блоки, все интеграции). Но по умолчанию они могут быть **выключены** или включаться через явную конфигурацию. 

Если клиенту не нужны "Кейсы", вы **не удаляете** код кейсов из `mu-plugins/agency-b2b-core`. Вы просто "не нажимаете кнопку включения" для этого клиента. Код остается лежать мертвым грузом (в PHP это не страшно, он не потребляет ресурсы, если не выполняется), а Git спокойно обновляет эти файлы в будущем без конфликтов.

### Как реализовать Feature Toggles в WordPress?

Есть два отличных паттерна для вашего `agency-b2b-core`:

#### Вариант 1: Через `add_theme_support` (Рекомендуемый для WP)

Тема клиента (или отдельный плагин-конфигуратор) решает, какие фичи ядра ей нужны. 

В клиентской теме (`functions.php`):
```php
// Включаем нужные фичи ядра для ЭТОГО клиента
add_action('after_setup_theme', function() {
    add_theme_support('agency-cpt-services');
    add_theme_support('agency-integration-amocrm');
    
    // add_theme_support('agency-cpt-cases'); // Клиенту не нужны кейсы — мы просто не просим ядро их регистрировать
});
```

В вашем ядре (`mu-plugins/agency-b2b-core/core.php`):
```php
// Регистрируем CPT только если тема явно запросила эту фичу
add_action('init', function() {
    if (current_theme_supports('agency-cpt-cases')) {
        require_once __DIR__ . '/includes/cpt-cases.php';
    }
    
    if (current_theme_supports('agency-cpt-services')) {
        require_once __DIR__ . '/includes/cpt-services.php';
    }
});
```

**Результат:** Код `cpt-cases.php` физически существует у клиента. Обновления `upstream` прилетают без проблем. Но на сайте клиента кейсов нет, так как условие `current_theme_supports` вернуло `false`.

#### Вариант 2: Через константы в `wp-config.php` (Для системных настроек)

Подходит для жестких системных интеграций (например, отключить WooCommerce логику на уровне сервера).

В `wp-config.php` клиента:
```php
define('AGENCY_ENABLE_CASES', false);
```

В ядре `agency-b2b-core`:
```php
if (defined('AGENCY_ENABLE_CASES') && AGENCY_ENABLE_CASES === true) {
    require_once __DIR__ . '/includes/cpt-cases.php';
}
```

## Идеальный процесс разработки для Агентства

1. **Развитие Бойлерплейта (`upstream`):**
   - Вы придумываете новые фичи (новые CPT, блоки, компоненты).
   - Все новые фичи сразу оборачиваются в условия проверки (Feature Toggles).
   - Вы коммитите это в главный репозиторий.

2. **Старт нового проекта (`origin`):**
   - Вы делаете `git clone` бойлерплейта.
   - Настраиваете Git-связи (`git remote rename origin upstream` и `git remote add origin git@github.com:client/repo.git`).
   - Вы заходите в конфигурацию и **включаете** только то, что нужно этому конкретному клиенту.

3. **Обновление старого клиента (Магия без конфликтов):**
   - Прошел год. Вы делаете `git fetch upstream && git merge upstream/main` в проекте клиента.
   - Так как вы **не удаляли** файлы ядра и **не меняли** их напрямую у клиента, Git производит слияние без единого конфликта.
   - Если в обновлении приехала новая фича (например, "Отзывы"), она по умолчанию скрыта за Feature Toggle. Сайт клиента продолжает работать как часы. Если клиент вдруг захочет отзывы, вы просто добавляете `add_theme_support('agency-cpt-reviews')`.

## Итог: Правило "Святого Ядра"

- **Никогда** не правьте файлы `agency-b2b-core` в проекте клиента (делайте это в бойлерплейте и тяните апдейт).
- **Никогда** не удаляйте файлы из `agency-b2b-core` в проекте клиента.
- Если клиенту нужна уникальная логика поверх ядра — используйте хуки (`do_action`, `apply_filters`) и пишите их в отдельном клиентском плагине (`mu-plugins/client-custom-logic.php`).
- Если клиенту не нужна базовая логика — отключайте ее флагами (Toggles), а не кнопкой Delete.

## Как выстроить процесс без оверинжинеринга на практике

1. **Разделяй и властвуй на уровне папок:**
   - `mu-plugins/agency-b2b-core/` — "Святая земля" бойлерплейта. Обновляется из Git. Читать можно, изменять нельзя.
   - `mu-plugins/client-core.php` — Земля клиента. Здесь пишется любая уникальная бизнес-логика клиента, создаются его кастомные CPT, переопределяются настройки через хуки.

2. **Фильтры вместо Delete (Удаление полей):**
   Если вам нужно убрать мета-поле (например, поле "Бюджет" у Кейсов), не удаляйте его из `agency-b2b-core`. Используйте фильтры WordPress в клиентском плагине:
   ```php
   // В mu-plugins/client-core.php:
   add_filter('agency_cases_meta_fields', function($fields) {
       // Отфильтровываем поле по имени или ключу
       return array_filter($fields, fn($f) => $f->get_name() !== 'budget');
   });
   ```

3. **Сознательный отказ от обновлений UI:**
   Как только вы начали верстать уникальный дизайн для клиента, визуальный слой этого проекта (тема) больше не нуждается в обновлениях "универсальных" стилей из бойлерплейта. Вытягивайте из `upstream` (через `git cherry-pick`) только обновления бэкенда (`agency-b2b-core`) и абстрактной PHP/JS логики, но не трогайте уникальную CSS верстку клиента.
