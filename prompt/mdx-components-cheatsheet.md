# Шпаргалка по MDX-компонентам Fumadocs (на основе текущей базы)

Проанализировав папку `content/docs/military`, я собрал все доступные React/MDX компоненты, которые уже точно работают в твоей сборке Fumadocs. Мы будем использовать **только их**, чтобы не "галлюцинировать" компонентами, которых нет в проекте.

## Доступные компоненты и их синтаксис

### 1. Callout (Сноски/Предупреждения)
Используется для выделения важной информации, подсказок или предупреждений.

```mdx
<Callout type="info">
  Это информационная сноска. Отлично подойдет для описания того, что файл ядра трогать нельзя.
</Callout>

<Callout type="warning">
  Предупреждение! Если вы удалите этот файл, Git Upstream при слиянии выдаст конфликт.
</Callout>
```

### 2. Tabs (Вкладки)
Отлично подходит для показа разницы в коде. Например, как было "до" и как надо делать "сейчас", или переключение между `upstream` и `origin`.

```mdx
<Tabs items={['В ядре (mu-plugins)', 'В теме (functions.php)']}>
  <Tab value="В ядре (mu-plugins)">
    ```php
    // Код ядра (не трогать!)
    ```
  </Tab>
  <Tab value="В теме (functions.php)">
    ```php
    // Подключаем фичу через add_theme_support
    add_theme_support('agency-cpt-services');
    ```
  </Tab>
</Tabs>
```

### 3. Accordions (Аккордеоны/Спойлеры)
Используется для скрытия длинных текстов, FAQ или дополнительных деталей, которые не обязательны к прочтению с первого раза.

```mdx
<Accordions type="single">
  <Accordion title="Как работает Feature Toggle под капотом?">
    Здесь можно описать глубокую техническую часть проверки констант и функций `current_theme_supports()`.
  </Accordion>
  <Accordion title="Что делать при конфликте слияния?">
    Пошаговая инструкция разрешения конфликтов.
  </Accordion>
</Accordions>
```

### 4. InlineTOC (Оглавление)
Встречается в файле `index.mdx` для рендера оглавления прямо в тексте (помимо бокового сайдбара).

```mdx
<InlineTOC items={toc}>Структура страницы</InlineTOC>
```

---

## Итог для Агента
При написании новых MDX файлов для документации Boilerplate, **запрещено** выдумывать компоненты (например, `<Card>`, `<Stepper>`, `<Files>` и т.д.), если мы не уверены, что они импортированы в конфигурации Fumadocs. Используем только:
- `<Callout>`
- `<Tabs>` / `<Tab>`
- `<Accordions>` / `<Accordion>`
- `<InlineTOC>`
- И стандартный Markdown (таблицы, списки, блоки кода, цитаты).
